# 🧱 PHASE 1 — STORAGE ENGINE (FOUNDATION)

This must be rock solid.

---

## 1️⃣ Page Layout Design

### Define:

```c
struct PageHeader {
    uint32_t page_id;
    uint16_t free_space_offset;
    uint16_t slot_count;
}
```

Then implement **slotted page layout**:

```
| Header | Free Space | Records | Slot Directory |
```

Each slot:

```c
struct Slot {
    uint16_t offset;
    uint16_t length;
}
```

### Why?

* Allows variable-length records
* Allows delete without shifting whole page
* Supports vacuum later

---

## 2️⃣ File Manager

Responsible for:

* Opening DB file
* Reading page N
* Writing page N
* Allocating new page

APIs:

```c
Page read_page(PageID id);
void write_page(PageID id, Page*);
PageID allocate_page();
```

It should:

* Use `pread` / `pwrite`
* Be page-size aligned
* Never cache anything

---

## 3️⃣ Buffer Pool

This is your first “real systems” component.

Responsibilities:

* Cache pages in memory
* Pin/unpin pages
* Track dirty pages
* Eviction (LRU or clock)

Core objects:

```c
struct Frame {
    Page page;
    bool dirty;
    int pin_count;
}
```

Buffer Manager must:

* Lookup page in hash table
* If not found → evict victim
* Write dirty page before eviction

---

## 4️⃣ Record Manager

Responsibilities:

* Insert record
* Delete record
* Update record
* Fetch record

Define:

```c
struct RID {
    PageID page_id;
    uint16_t slot_id;
}
```

Record insertion:

* Find page with space
* Write data
* Add slot entry
* Return RID

---

# 🌲 PHASE 2 — INDEXES (B+ TREE)

You already touched this conceptually.

---

## 1️⃣ Index Page Types

```c
enum PageType {
    INTERNAL,
    LEAF
};
```

Leaf:

* key → RID

Internal:

* key → child_page_id

---

## 2️⃣ B+Tree Operations

You must implement:

* search(key)
* insert(key, rid)
* split leaf
* split internal
* root split
* delete (later)

Key concept:

* All leaves linked (range scan support)

---

## 3️⃣ Index Manager

Maps:

```
index_name → root_page_id
```

Stores metadata in system catalog.

---

# 🔐 PHASE 3 — TRANSACTION SYSTEM

Now things become real.

---

## 1️⃣ WAL (Write Ahead Log)

Log record types:

```c
enum LogType {
    INSERT,
    DELETE,
    UPDATE,
    BEGIN,
    COMMIT,
    ABORT
};
```

Log record:

```c
struct LogRecord {
    LSN lsn;
    TxnID txn_id;
    LogType type;
    PageID page_id;
    before_image;
    after_image;
}
```

Rules:

* Log must flush before dirty page flush
* Every page stores its latest LSN

---

## 2️⃣ Log Manager

Responsibilities:

* Append log record
* Assign LSN
* Flush log buffer

APIs:

```c
LSN append_log(LogRecord*);
void flush_upto(LSN);
```

---

## 3️⃣ Recovery (ARIES-lite)

Implement 3 phases:

1. Analysis
2. Redo
3. Undo

You can simplify:

* No fuzzy checkpoints at first
* Just full scan recovery

---

## 4️⃣ Lock Manager

Implement:

* Shared locks (S)
* Exclusive locks (X)

Use:

```c
map<ResourceID, LockQueue>
```

Support:

* Lock upgrade
* Deadlock detection (wait-for graph)

---

## 5️⃣ Transaction Manager

Manages:

```c
struct Transaction {
    TxnID id;
    state;
    list<Lock>;
}
```

Supports:

* begin
* commit
* abort

---

# 🧠 PHASE 4 — QUERY PROCESSOR

Now we move to SQL layer.

---

# 1️⃣ LEXER (Tokenizer)

Input:

```
SELECT name FROM employees WHERE id = 5;
```

Token types:

* KEYWORD
* IDENTIFIER
* NUMBER
* STRING
* OPERATOR

Build DFA-based tokenizer.

---

# 2️⃣ PARSER (AST Builder)

Define AST nodes:

```c
struct SelectStmt {
    vector<Expr> projection;
    TableRef from;
    Expr where;
}
```

Implement recursive descent parser.

---

# 3️⃣ BINDER

Resolves:

* Table names
* Column names
* Types

Turns AST into:

```
Logical Plan
```

---

# 4️⃣ LOGICAL PLAN NODES

Examples:

```
LogicalScan
LogicalFilter
LogicalProject
LogicalJoin
LogicalAggregate
```

---

# 5️⃣ OPTIMIZER

Start simple:

* Rule-based optimization
* Predicate pushdown
* Index selection
* Projection pruning

Later:

* Cost-based optimizer
* Cardinality estimation

---

# ⚙️ PHASE 5 — EXECUTION ENGINE

Turn logical plan into physical operators.

---

## Physical Operators

Implement iterator model:

```c
class Executor {
    virtual void Init();
    virtual Tuple Next();
}
```

Operators:

* SeqScanExecutor
* IndexScanExecutor
* FilterExecutor
* ProjectionExecutor
* NestedLoopJoinExecutor
* HashJoinExecutor

Each pulls tuples from child.

---

# 🔄 Full Query Flow

User runs:

```
SELECT * FROM employees WHERE id = 10;
```

Flow:

1. Tokenize
2. Parse → AST
3. Bind → Logical Plan
4. Optimize → Physical Plan
5. Executor:

   * Begin transaction
   * Acquire locks
   * Fetch page from buffer
   * Return tuple
6. Commit
